HOW TO READ THIS PLAN (proof of index usage)
============================================================================
1. "Bitmap Index Scan on companies_municipio_ativa_idx" — the territory filter
   (municipio_id = ANY(...)) is served by the PARTIAL btree index that only
   covers situacao_cadastral='ativa'. No seq scan over 1.6M rows.
2. "Filter: (cnae_divisao = ANY('{45,46,47}'))" prunes candidates down to the
   CNAE-relevant set on the heap. companies_divisao_ativa_idx is available; the
   planner keeps the (more selective) municipio index as the driver and applies
   the division predicate as a recheck — both are partial-ativa indexes.
3. "Nested Loop Anti Join" against company_relationships implements the
   NOT EXISTS exclusion ("already in my funnel") using the tenant index on
   (org_id, ...). The global companies table is never written, only read.
4. Sort is in-memory quicksort (work_mem=64MB) — no disk spill.
5. Execution Time ~0.35s for the WORST case: 3 dense metro municipios
   (~101k scored candidates). Typical rep territory (1-2 smaller municipios +
   narrower CNAE) scores far fewer rows and returns in tens of ms.

RADIUS MODE (ST_DWithin) uses "Bitmap Index Scan on companies_geom_ativa_idx"
(PARTIAL GIST) — verified separately, ~0.31s within 50km of São Paulo.

Production notes baked into docker-compose / app:
- work_mem is raised per-session for the recommendation query.
- The query is a SINGLE statement (no N+1), paginated by LIMIT/OFFSET.
